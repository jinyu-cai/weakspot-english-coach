# UI 重设计本地预览说明

> 当前状态：仅用于本地预览与设计评审。工作分支为 `ui/warm-onboarding-preview`，尚未部署、尚未合并到 `main`，也不会在用户确认前替换线上界面。

## 1. 本地预览分支目标

本分支用于探索一版更温馨、简洁、清晰的 WeakSpot UI，让第一次使用的用户不需要理解产品内部结构，也能快速完成第一项有效学习任务。

重设计只调整信息架构、视觉层级、说明文案和交互呈现，不减少现有功能，不改变已确认的数据生命周期，也不为 Notebook、History 等核心资料增加固定数量限制。

目标包括：

- 让新用户一眼看懂“现在可以做什么”以及推荐从哪里开始。
- 降低 12 个同级主导航入口带来的认知负担，同时保留所有入口。
- 用温暖但克制的颜色、圆角、留白和鼓励性文案建立陪伴感，避免冷酷的管理后台观感。
- 将模型、Memory 检索、掌握阈值等高级能力放到合适层级，不让它们妨碍核心学习流程。
- 保证手机、平板和桌面端都能完整显示，不依赖大屏幕。
- 在本地展示、测试和评审通过后，再由用户决定是否采用、合并和部署。

## 2. 设计原则

1. **先行动，后解释**：页面先给出清晰的主要操作，再按需展示原理、统计和高级设置。
2. **功能完整，层级精简**：可以合并视觉入口、分组导航或折叠高级内容，但不能删除任何能力。
3. **温馨而不幼稚**：保持暖色背景、柔和强调色和适度的猫头鹰品牌元素，同时控制装饰数量。
4. **一个页面一个主任务**：每个页面只突出一个主要按钮，其余操作保持可发现但降低视觉权重。
5. **循序渐进**：复杂表单和专业信息采用步骤、摘要、展开详情或“高级”区域呈现。
6. **数据状态说人话**：优先使用“正在学习”“以前学过”“建议复习”等用户语言，技术状态作为补充。
7. **破坏性操作可确认**：永久删除、遗忘或移除权限必须清楚说明影响范围，避免误操作。
8. **响应式优先**：窄屏下允许纵向排列和换行，禁止关键文字、模型名、技能名和操作按钮被裁切。
9. **无隐性数据上限**：摘要可以精简，但必须明确标注并提供“查看全部”；完整资料页不设置固定显示上限。
10. **可访问与可理解**：保留键盘操作、焦点状态、语义标签、颜色之外的状态提示以及中英双语。

## 3. 导航分组方案

现有 12 个主导航入口全部保留，但从同级长列表调整为面向任务的分组：

### 开始学习

- **诊断** `/`：分析真实英文，建立或更新薄弱项。
- **对话** `/chat`：进行文字或语音交流练习。
- **练习** `/practice`：进行混合或指定技能的自由定向训练。

### 学习内容

- **输入学习** `/input`：把影视、阅读、播客或真实材料转为学习内容。
- **计划** `/plan`：生成和完成个性化 7 天计划。
- **导入** `/import`：导入并分析 ChatGPT 对话历史。

### 我的进步

- **仪表盘** `/dashboard`：查看英语水平和薄弱项模型。
- **每日进步** `/stats`：查看连续学习、活动和成就。

### 学习资料

- **笔记本** `/notebook`：查看系统自动收集的当前与以前笔记。
- **历史** `/history`：查看所有提交、纠错及关联记录。

### 你的教练

- **记忆** `/memory`：查看和管理教练的跨会话记忆；高级指标默认降低视觉权重。

### 管理

- **管理** `/admin`：仅 owner 可见，用于维护成员权限。

`/login` 和 `/plan/practice` 保持为流程页面，不占用主导航位置。

## 5. 全局功能保留

以下能力跨页面存在，重设计不得遗漏：

- 中英文切换及现有本地化内容。
- 明亮/深色主题和现有配色方案。
- 移动端导航抽屉、桌面端导航和完整响应式布局。
- 登录、退出、头像，以及 owner/member 权限状态。
- 服务器 Deep/Fast 模型选择。
- 自定义 OpenAI 兼容 Base URL、深度模型、快速模型和 API Key（BYOK）。
- 模型目录加载失败后的重试与恢复路径。
- 语音会话开始、连接、静音、转录保存、保存失败重试，以及语音进行中阻止离页。
- Toast、加载、空状态、错误重试和部分成功提示。
- 诊断、对话、练习、导入对 Profile、弱点模型、Memory、Notebook 和 History 的联动。
- 用户主动删除 History 时的后端永久删除和证据撤回语义。
- 系统判定薄弱项解决时保留以前笔记，以及后续复发时自动恢复为当前笔记的语义。

## 6. 隐性摘要限制审计与本原型处理

审计发现以下页面曾在前端使用固定截取。它们不是后端数据上限，但会造成“只有前几条”的误解。本地原型已经移除这些静默截取，现有滚动容器会渲染接口返回的完整集合：

| 位置 | 原先固定截取 | 本地原型处理 |
| --- | ---: | --- |
| Chat 最近会话 | 5 条 | 已移除 `slice(0, 5)`，显示接口返回的全部会话 |
| Dashboard 最弱技能 | 5 项 | 已移除 `slice(0, 5)`，显示全部技能 |
| Dashboard 最近错误 | 5 条 | 已移除 `slice(0, 5)`，显示全部最近错误 |
| Import 对话预览 | 8 条 | 已移除 `slice(0, 8)`；滚动预览完整选择集合，分析仍支持“全部” |
| Memory 召回轨迹 | 8 条 | 已移除 `slice(0, 8)`，滚动区域显示全部轨迹 |
| Memory 每条轨迹的已选记忆 | 3 条 | 已移除 `slice(0, 3)`，显示完整选中项 |

实现要求：

- 不得把摘要数量误写为数据总量。
- 如果后续因数据量增长改用“查看全部”“加载更多”或分页，它们必须真实访问完整数据，不能只扩大固定常量。
- Notebook 和 History 的核心列表保持无固定总量上限，并正确处理后端分页。
- 长列表可以通过后端游标分页、渐进加载或虚拟滚动改善性能，但用户必须最终能访问全部记录。
- Plan 的 7 天、Stats 的 7 天统计窗和 Practice 的每轮 4 题属于明确产品结构，需要清楚标注，不视为隐性资料上限。

## 7. 此次原型范围

本地原型包括：

- 全局应用外壳、导航分组、顶栏层级和移动端导航体验。
- 更温暖、克制的颜色、圆角、留白、卡片、按钮和状态反馈。
- 新用户首次进入时的起步引导和主要行动层级。
- 14 个既有路由的入口可达性和功能完整性核对。
- Input、Import、Memory、Notebook 和 History 的下一阶段渐进展示方案与功能保留清单。
- 语言、明暗模式和四套配色合并为一个偏好入口；完整模型与 BYOK 设置继续保留在顶栏。
- 窄屏、长文字、完整技能名和完整模型名的响应式处理。
- 清除 Chat、Dashboard、Import、Memory 的固定前端摘要截取。
- 本地 TypeScript、Lint、构建及必要交互检查。

本轮为了先让用户确认设计方向，代码层面的深度改版集中在全局应用外壳、视觉系统和诊断起步页；其余页面完整保留原有交互，并继承新的卡片、按钮、输入框、Tabs、配色和响应式规则。用户确认方向后，再按本文件的功能矩阵逐页调整信息层级。

此次原型不包括：

- 后端 API、数据库表、数据保留策略或弱点算法的重新设计。
- Oracle、Ali、Cloudflare、Vercel 或域名配置变更。
- 线上部署、生产流量切换或数据迁移。
- 在未获得用户确认前合并到 `main`。

## 8. 评审与发布边界

评审时应确认：

- 第一次进入的用户是否能在短时间内选择并开始第一项学习任务。
- 导航更精简后，所有原有入口和操作是否仍然可找到。
- 手机和小屏幕是否完整显示技能、模型、筛选器、表格、图表和操作按钮。
- Notebook、History 及其他完整资料是否不存在固定总量上限。
- 高级功能是否仍可用，但不会干扰普通用户完成核心学习流程。
- 界面是否温暖、清晰、安静，而不是冷酷、拥挤或过度装饰。

只有在用户查看本地预览并明确决定采用后，才可以进入提交、合并和部署流程。当前文档与原型均不代表上线批准。
